Vision Statement

Our app sends notifications. The idea is based on the concept that if you can unpack notifications from the app package you can send notifications as a developer or receive them as a user without downloading an app. In addition you can bring a singular paradigm for turning up or down the frequency of notifications received.

The idea is the create a relay app where users can sign up for notifications from various 3rd party notifiers. On a developer side we can include many of the things that make their app relevant for their users already, or their potential users. For instance we can request location access at a macro level and give access to it for notification senders. The idea is that our server will intercept notifications from the 3rd part developers, vet them to make sure we reduce spam and send them to users. We can also allow for algorithmic curating, such as a parring the number of notifications a user gets if they don't click on many of them.

In addition we would love to have the ability to send notifications between users/friends and to allow for interaction with notifications with iOS's new interactive notifications.

Feature List

  1. Our Framework provides an app through which users can sign up for our services.
  2. Our Framework provides assignment of API keys to third parties so third-parties can use our services.
  3. Our Framework supports two different types of notifications UserNotifications and DeveloperNotifications.
  4. Our Framework allows both notification types to be pushed to the receiving users' phones and show up in the receiving users' apps.
  5. Through the assigned API key, our Framework allows third party developers to send a DeveloperNotification to users by making a JSON request to our servers.
  6. Our Framework requires DeveloperNotification to be labeled with at least one hashtag and will forward the notification to the user base of each hashtag.
  7. Our Framework allows third parties to send notifications with different level of priority.
  8. Through the provided app, our framework allows users to sign up for notificiation from certain hashtags
  9. Through the provided app, our Framework allows users to forward (or send) notification, and so creating a UserNotification, to other "trusted" users.
  10. Through the provided app, our Framework allows users to add users as "trusted" as well as delete users from their "trusted" list through.
  11. Through the provided app, our Framework allows users to limit the number of notification based on priority levels and/or location ranges.
  12. Through the provided app, our Framework allows users to modify notification sound per notification.
  13. Our Framework supports tracking of whether or not a user opened notifications.

Expanded Features

  1. Our Framework would allow specific sponsor notifications
  2. Our Framework would support combined notifications of distinct third party users
  3. Our Framework would allow third parties to send notification if users trip a geo-wire
  4. Through the provided app, our Framework allows users to add additional text or buttons to a forwarded notification when creating a UserNotification.
  5. Our Framework would display opening/closing statistics to third parties which third parties could use to improve their notifications.

Domain Model

Our application's front-end will be in the form of an iOS application. Our server will serve as a front-end for developers from which they can create accounts, create and edit subscription lists, and view statistics for their subscription lists.

Below is a UML Class diagram for our developer-centric server.

Deployment Diagram

Actors

Developer

  • Create one or many subscription lists
  • Push a notification to subscribed users for any list

Common User

  • Recieve notifications from an iOS app
  • Subscribe to any list they want
  • Push simple notifications to other users
  • Filter incoming notifications with personal settings

Use-cases

Title

Positing a notification

Primary Actor

Developer, posts a notification to his user base

Second Actor

Users, the users that recieve the developer's notification

Pre-condition

For the notification to be pushed the developer should have an event he send users notifications for.

If that event happens then he notifies the users of its occurrence.

Main Path

  1. Developer pushes notification to our server
  2. Verify the API key. If key is incorrect then go to alternate path #2
  3. We update each of the relevant users profiles to include the pushed notification extended
  4. If the priority of the notification is high enough we may push a system wide notification to the iOS device. (OS widenotificationn rather than a listed notification inside the application

Alternate Path #1

  1. A user posts a notification to another user
  2. Our server takes this request and dispatches this out if allowed
  3. If not allowed it will notify the original user of this fact.

Alternate Path #2

  1. Do not allow Developer to push notification

Title

Receive a notification

Primary Actor

User

Secondary Actor

Developer

Goal

Allow the ability for users to receive notifications from Developers or other Users

Pre-Condition

A notification must be sent or posted from a developer or initial user in order for other users to receive notifications.

Main Path

  1. user subscribes to a developer's subscription
  2. notification is created by a developer
  3. notification is pushed to the developers subscription list
  4. notification is sent to all of the subscriptions subscribers, i.e., other users
  5. notification is received by a user

Alternate Path

  1. notification is created by a user
  2. notification is pushed to another user
  3. notification is received by a user

Title

Change Setting

Primary Actor

User

Secondary Actor

Developer

Goal

A user will be able to adjust his/her application preferences

Pre-Condition

A user wants to change the way he/she interacts with the application

Main Path

  1. User is wants to change preferences
  2. User opens up the settings tab in the application
  3. User can change settings, which may include:
    1. frequency of notifications
    2. only receive a certain priority level
    3. unsubscribe from certain subscriptions
  4. If the user unsubscribes, he/she will be removed from appropriate developers subscription list.

Title

Developer Sign Up

Primary Actor

Developer

Secondary Actor

Us, Allowing developer sign up

Goal

Is to get a developer an API key for his new list, allowing Users to sign up to receive his notifications.

Pre-Condition

No real pre conditions except a willingness to send notifications to subscribed users

Main Path

  1. Developer visits our website and fills out the sign up sheet, giving us an email and a requested stream title
  2. After verifying email and making sure the stream title is not taken already we give them an api key to send their notifications too
  3. We also add their stream title to the subscribable list for users
  4. They can add a sub title to their stream, to provide more information about what they provide for users
  5. from there on out they can send notifications to all subscribed users by sending an API call.

Title

List Received Notification

Primary Actor

User

Goal

A User wants to see a list of all of the notifications he/she received

Pre-Condition

No real pre-condition

Main Path

  1. User goes to Settings section
  2. User requests to see all of the Notifications he/she received
  3. Return a list of all Notifications received

Alternate Path

  1. If no Notifications have been received User gets an empty list
  2. User gets a message saying "no notifications received"

Title

Un/Subscribe

Primary Actor

User

Secondary Actor

Developer

Goal

Change a subscriptions list of subscribers if a new User subscribes or unsubscribes

Pre-Condition

The subscription being Un/Subscribed must exist

Main Path

  1. A user Subscribes to a Developers Subscription
  2. The User is added to the Developer's Subscription's list
  3. The Developer will send Notifications to this User from now on

Alternate Path

  1. A user unSubscribes to a Developers Subscription
  2. The User is removed to the Developer's Subscription's list
  3. The Developer will no longer send Notifications to this User

Architecture

We will use iOS front end for user interaction and most likely a web front end for Developer interaction. Django/JSON something of that sort, the developer site will be simple since it will just be a sign up sheet to get an API token. On the backend we will use a server and a database. The server will receive and send notifications with the database storing user information.
